# Claude Code サブエージェント活用マニュアル
## ～複数のAIに役割を与えて「チーム」として運用する方法～
### （プライベート：メダカ図鑑作成プロジェクト版）

---

## 第1章 サブエージェントとは？

### 1-1. ひとことで言うと

**「メインのClaude（監督）が、専門の部下AI（選手）に仕事を割り振る仕組み」** です。

サッカーチームに例えると：

| サッカーチーム | Claude Code |
|---|---|
| 監督 | メインのClaude（あなたが直接会話する相手） |
| フォワード（攻撃担当） | コンテンツ作成担当のサブエージェント |
| ディフェンダー（守備担当） | レビュー・チェック担当のサブエージェント |
| スカウト（偵察担当） | 調査・検索担当のサブエージェント |

監督（メインのClaude）は自分で全部やらずに、**得意な選手に仕事を任せて、結果だけを受け取ります**。

### 1-2. なぜチームにすると良いのか？（3つのメリット）

1. **メモリ（コンテキスト）の節約**
   - AIには「一度に覚えられる量」に限界があります。
   - サブエージェントは**別の頭脳**で動くので、大量のファイルを読む作業を任せても、メインの会話のメモリを消費しません。
   - 例え：監督がノート1冊しか持てないなら、調査は部下に任せて「結論だけ」報告してもらう方が効率的。

2. **専門特化で精度アップ**
   - 各サブエージェントに「あなたは〇〇の専門家です」のような**専用の指示書**を持たせられます。
   - 余計な情報がない分、その仕事に集中できます。

3. **並列作業で時間短縮**
   - 複数のサブエージェントを**同時に**動かせます。
   - 例え：3人で手分けして調べれば、1人で調べるより3倍速い。

### 1-3. 注意点（デメリットも知っておく）

- **コストがかかる**：サブエージェント1体ごとにAIの利用量（トークン）を消費します。むやみに増やさない。
- **会話の記憶を引き継がない**：サブエージェントは毎回「まっさらな状態」で起動します。**指示は具体的に書く**必要があります（ファイルパス、前提条件など）。
- **結果は報告書だけ**：サブエージェントの作業過程は見えず、最後の報告だけがメインのClaudeに返ります。

## 第2章 サブエージェントの作り方

### 2-1. 置き場所

サブエージェントの定義は **Markdownファイル（.md）** で作り、以下のフォルダに置きます。

| 置き場所 | 適用範囲 |
|---|---|
| `~/.claude/agents/`（例：`C:\Users\0300036\.claude\agents\`） | **全プロジェクト共通**で使える |
| `プロジェクトルート/.claude/agents/`（例：`C:\Users\0300036\Documents\メダカ図鑑\.claude\agents\`） | **そのプロジェクト専用** |

※ 同じ名前がある場合は、**プロジェクト側が優先**されます。
※ **このプロジェクトではプロジェクト専用側に配置済み**です。プライベートの部下AIが仕事のセッションに出てくることはありません。

### 2-2. ファイルの書き方（基本形）

ファイルの先頭に「フロントマター」と呼ばれる設定欄（`---`で囲む部分）を書き、その下に指示書（システムプロンプト）を書きます。

```markdown
---
name: エージェント名（英小文字とハイフン）
description: どんな時に使うエージェントかの説明（これを見てメインClaudeが呼ぶか判断する）
tools: 使わせるツール（省略すると全ツール使用可）
model: sonnet などモデル指定（省略するとメインと同じ）
---

ここから下が、このサブエージェントへの指示書です。
「あなたは〇〇の専門家です」から始めて、
役割・手順・出力形式を具体的に書きます。
```

### 2-3. 各項目の意味

| 項目 | 必須 | 説明 |
|---|---|---|
| `name` | ○ | 呼び出し名。例：`medaka-researcher` |
| `description` | ○ | **一番重要**。メインClaudeはこの説明文を読んで「この仕事はこの部下に任せよう」と自動判断する。「〜の時に使う」と具体的に書く |
| `tools` | － | 安全のため必要最小限に絞るのがおすすめ。例：調査係なら `Read, Grep, Glob, WebSearch, WebFetch`（ファイル書き換え不可になる） |
| `model` | － | 単純作業は `haiku`（速い・安い）、難しい作業は `opus` など使い分け可能 |

### 2-4. 対話式で作る方法

ターミナルのClaude Codeで `/agents` と入力すると、対話形式でエージェントの作成・編集ができます（ファイルを手書きしなくてもOK）。

## 第3章 実践：メダカ図鑑チームの構成（配置済み）

このプロジェクトには、以下の3体が `C:\Users\0300036\Documents\メダカ図鑑\.claude\agents\` に**配置済み**です。

> **チームの全体像**
> - 品種調査係（medaka-researcher）：品種の特徴・由来・飼育情報を調べて素材を集める
> - 図鑑ページ作成係（zukan-writer）：素材をテンプレートに沿って図鑑ページに仕上げる
> - 校正係（zukan-checker）：表記ゆれ・記入漏れがないか最終チェックする

### 3-1. 品種調査係：`medaka-researcher.md`

```markdown
---
name: medaka-researcher
description: メダカの品種（幹之、楊貴妃、オロチなど）の特徴・由来・飼育情報を調査する時に使う。Web検索と手元メモの読み取りのみ行い、ファイルの変更は行わない。
tools: Read, Grep, Glob, WebSearch, WebFetch
---

あなたはメダカ（改良メダカ）の品種調査の専門家です。

## 役割
- 指定された品種について、次の項目を調査してまとめる
  1. 品種名（正式名・別名）
  2. 見た目の特徴（体色・ラメ・ヒレの形など）
  3. 作出の由来（元になった品種・作出者・発表年がわかれば）
  4. 飼育難易度（初心者向けか、注意点）
  5. 繁殖のポイント（固定率＝親と同じ姿の子が生まれる割合、の目安があれば）

## ルール
- ファイルの変更・作成は行わない（調査と報告のみ）
- Web情報は複数のサイトで突き合わせ、情報源のURLを必ず添える
- 確認できなかった項目は「不明」と正直に報告する（推測で埋めない）
- 報告は上記1～5の項目順で、箇条書きで簡潔に
```

### 3-2. 図鑑ページ作成係：`zukan-writer.md`

```markdown
---
name: zukan-writer
description: メダカ図鑑の品種ページ（Markdown）を新規作成・修正する時に使う。調査済みの素材をテンプレートに流し込んで整形する。
tools: Read, Write, Edit, Glob
---

あなたはメダカ図鑑のページ作成担当です。

## 役割
- 渡された調査結果をもとに、1品種＝1ファイルの図鑑ページを作成する
- 保存先：C:\Users\0300036\Documents\メダカ図鑑\図鑑\
- ファイル名：品種名.md（例：幹之.md）

## ページテンプレート（この構成を厳守）
# 品種名
## 基本情報
| 項目 | 内容 |
|---|---|
| 正式名・別名 | |
| 体色・特徴 | |
| 作出の由来 | |
| 飼育難易度 | ★1～5で表記 |
| 固定率の目安 | |
## 特徴の詳細
（見た目の魅力を2～3文で）
## 飼育のポイント
（箇条書き3項目以内）
## 写真
（写真ファイル名を記載。例：photos/幹之_01.jpg）
## 情報源
（調査係から受け取ったURLを列挙）

## ルール
- テンプレートにない見出しを勝手に追加しない
- 調査結果にない情報を創作しない。空欄は「調査中」と記載する
- 品種名の表記を統一する（みゆき→幹之、ようきひ→楊貴妃）
- 語り口は「です・ます調」で統一する
```

### 3-3. 校正係：`zukan-checker.md`

```markdown
---
name: zukan-checker
description: 完成したメダカ図鑑ページの最終チェックが必要な時に使う。表記ゆれ・記入漏れ・テンプレート崩れを確認する。ファイルの変更は行わない。
tools: Read, Grep, Glob
---

あなたはメダカ図鑑の校正担当です。

## チェック観点
1. テンプレート：見出し構成が規定どおりか（基本情報→特徴の詳細→飼育のポイント→写真→情報源）
2. 表記ゆれ：品種名の揺れがないか（例：「みゆき／ミユキ」→「幹之」に統一、「ようきひ」→「楊貴妃」に統一）
3. 記入漏れ：空欄のまま放置されていないか（空欄は「調査中」と書かれているか）
4. 数値表記：飼育難易度が「★1～5」形式か
5. 情報源：URLが1件以上記載されているか

## 報告形式
- 問題なし → 「合格」と一言
- 問題あり → 「ファイル名・箇所・問題点・修正案」を表形式で報告
- 修正は自分では行わず、報告のみ（修正は zukan-writer の仕事）
```

## 第4章 チームの動かし方（運用パターン）

**前提：Claude Codeは `C:\Users\0300036\Documents\メダカ図鑑\` フォルダで起動してください**（このフォルダのCLAUDE.mdとエージェントが読み込まれます）。

### 4-1. 呼び出し方は2通り

1. **自動委任**：普通に依頼するだけ。メインClaudeが `description` を見て、適切なサブエージェントに自動で振り分けます。
   > 例：「楊貴妃メダカの特徴を調べて」→ 自動で medaka-researcher が起動

2. **明示的に指名**：エージェント名を指定して確実に呼ぶ方法。
   > 例：「**zukan-writerエージェントを使って**、幹之のページを作って」
   > 例：「**zukan-checkerでチェックしてから**完成にして」

### 4-2. よく使う3つのチーム編成

#### パターンA：バケツリレー型（直列）
調査 → 作成 → チェック の流れ。品質が大事な成果物向け。

```
あなた「オロチメダカの図鑑ページを、チェックまで通して完成させて」
  ↓
① medaka-researcher が品種情報を調査
  ↓
② zukan-writer がテンプレートに沿ってページ作成
  ↓
③ zukan-checker が表記・記入漏れをチェック
  ↓
④ メインClaudeが合格版をあなたに納品
```

#### パターンB：手分け型（並列）
独立した調査を同時に走らせる。時間短縮向け。

```
あなた「幹之・楊貴妃・オロチの3品種をそれぞれ調べて」
  ↓
medaka-researcher ×3体が同時に調査
  ↓
メインClaudeが3つの報告をまとめて回答
```

#### パターンC：監督集約型
メインClaudeが計画を立て、各専門係に割り振り、最後に統合。大きな作業向け。

```
あなた「飼っている10品種ぶんの図鑑を丸ごと作って」
  ↓
メインClaude（監督）が作業を分解
  ├─ medaka-researcher：10品種の情報収集（並列）
  ├─ zukan-writer：ページを順次作成
  └─ zukan-checker：全ページの最終チェック
  ↓
監督が統合して「図鑑」フォルダに納品
```

### 4-3. うまく任せるコツ

- **指示は「新入社員への依頼メモ」のつもりで書く**
  - サブエージェントは会話の流れを知りません。「さっきの件」は通じない。
  - 悪い例：「あれを直して」
  - 良い例：「図鑑\幹之.md の飼育難易度を★3に修正して」
- **1体には1つの仕事**：役割を欲張らせない。「調査も執筆も校正もできる万能係」より、専門係を分ける方が精度が上がる。
- **descriptionを磨く**：自動委任がうまく働かない時は、`description` に「〜の時に使う」を具体的に書き足す。

## 第5章 よくある質問（FAQ）

**Q1. サブエージェント同士は直接会話できる？**
A. できません。必ずメインClaude（監督）を経由して情報をやり取りします。調査係の結果は監督が受け取り、監督が作成係に渡します。

**Q2. 何体まで作れる？**
A. 定義ファイルはいくつでも作れます。ただし同時に動かす数が増えるほどコスト（トークン消費）も増えるので、必要な分だけ動かすのが基本です。

**Q3. メインの会話が長くなってもサブエージェントに影響する？**
A. しません。サブエージェントは毎回まっさらな状態で起動するので、逆に「メインの会話を汚したくない大量調査」を任せるのに向いています。

**Q4. 権限（ファイル書き換えなど）が心配**
A. `tools` を絞れば安全です。調査係と校正係には書き込み系ツールを与えていないので、勝手にファイルを変更することはできません。書き込めるのは zukan-writer だけ、という役割設計そのものが安全装置になります。

**Q5. 作ったエージェントが呼ばれない**
A. まず「〇〇エージェントを使って」と明示指名で動作確認 → 動くなら `description` の書き方の問題なので、「どんな時に使うか」をより具体的に書き直してください。

**Q6. 仕事のClaude Codeに影響しない？**
A. しません。エージェント3体はこのプロジェクトの `.claude\agents\` に置いてあるので、このフォルダで起動した時だけ有効です。CLAUDE.mdもプロジェクト用が読み込まれます。
